# Серверный режим сортировки: не пересортировывать серверную страницу локально

**Дата:** 2026-06-12
**Статус:** Design (одобрен к реализации)
**Тип изменения:** fix поведения, без нового API. Изменение поведения для `pageSource`-таблиц
с активной сортировкой — порядок строк станет серверным (раньше перетирался локальной
пересортировкой); отметить в changelog 19.7.0 как fix с примечанием.
**Контекст батча:** пункт 1 фидбека от 2026-06-11 (Andrey Patsko); ветка `feat/19.7.0-feedback`,
пункты реализуются по одному, коммит на пункт.

## Проблема

Передача сортировки на сервер уже работает: клик по заголовку → `matSortChange` →
`sortTable()` → `ServerPageController.onSort()` → запрос с `sort` и сбросом на страницу 0.

Сломана другая половина: `initSortingDataAccessor()` безусловно привязывает
`tableDataSource.sort = matSort`. В серверном режиме это даёт:

1. **Мгновенную локальную пересортировку старой страницы** сразу после клика —
   визуальный «прыжок» до прихода серверного ответа.
2. **Перезапись серверного порядка**: каждое `tableDataSource.data = ...` прогоняет
   `_orderData`, и пришедшая страница пересортировывается локально по значениям
   `valueConverter`. Сервер сортирует по сырому полю (даты, числа, локаль, nulls,
   поля связанных сущностей), локально сортируются отформатированные строки —
   порядок расходится (классика: даты `dd.MM.yyyy` лексикографически).
3. `customSort` колонок конкурирует с сервером на тех же правах.

## Решение (зафиксировано с пользователем)

| Решение                                             | Выбор                                                                                                                                   |
|-----------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------|
| Откуда таблица узнаёт о серверной сортировке        | автоматически из серверной обвязки — `isServerWiring()` (`pageSource` ИЛИ `paginationCfg.mode: 'server'`); новых конфигов нет           |
| Кейс «серверная сортировка без серверной пагинации» | у потребителя отсутствует (ответ «a»); переключатель не закладываем — при появлении кейса ляжет в табличный `sortCfg` пункта 2          |
| Явный `sort: { mode: 'server' }` из пожелания       | отклонён: режим всегда выводим, явный конфиг создаёт невозможные комбинации; per-column `mode` некогерентен (один `MatSort` на таблицу) |
| Legacy manual (`paginatorState` без `mode`)         | поведение не меняется (локальная сортировка остаётся) — не ломаем существующих потребителей                                             |

## Реализация — `ngx-aur-mat-table.component.ts`

Одно место, `initSortingDataAccessor()`:

```ts
// Серверная обвязка: сортирует сервер — не привязываем MatSort к dataSource,
// иначе _orderData пересортировал бы серверную страницу по значениям valueConverter
// (зеркало initPaginator(), который по той же причине не привязывает пагинатор).
// Стрелки и matSortChange живут на директиве MatSort и от привязки не зависят.
const sort = this.isServerWiring() ? null : (this.matSort ?? null);
```

Остальное без изменений:

- `sortingDataAccessor` назначается как раньше — при `sort = null` инертен,
  нужен клиентскому режиму;
- `sortTable()` как и сейчас эмитит `(sort)` и дёргает `ServerPageController.onSort()`;
- существующий гвард `!==` на переприсваивание сохраняется.

## Поведение по режимам

| Обвязка | Локальная сортировка | Серверный запрос при клике |
|---|---|---|
| Клиентская (default) | да — как сейчас | — |
| `pageSource` | **нет** (фикс) | да — уже работает |
| `mode: 'server'` без `pageSource` (ручной режим, задел под пункт 5) | **нет** (фикс) | хост сам, через `(sort)` |
| Legacy manual (`paginatorState` без `mode`) | да — не трогаем | хост сам |

## Edge cases

- **In-flight**: после клика страница остаётся в старом порядке до прихода ответа
  (исчезает «прыжок»); стрелка уже показывает новое направление, `loadingChange` = true.
- **`customSort`** в серверном режиме инертен — дополнить JSDoc `SortConfig.customSort`.
- **`updateTimelineBounds()`** читает `matSort.active` с директивы, не с dataSource — не задет.
- **Локальный поиск/фильтры** по загруженной странице — поведение не меняется (вне скоупа).
- **`isServerWiring()`** уже существует и используется в `ngOnInit` для авто-инициализации
  `paginatorState` — новых хелперов не требуется.

## Тесты — новый `ngx-aur-mat-table-server-sort.spec.ts`

1. **Серверный порядок сохраняется**: сервер отвечает в порядке, который локальный
   компаратор перевернул бы (например `['b','a']` при `asc`) → в таблице ровно `['b','a']`.
2. **Запрос с сортировкой**: клик по заголовку → `pageSource` вызван с
   `{sort: {active, direction}, pageIndex: 0}`.
3. **Нет in-flight прыжка**: `pageSource` на `Subject`; до ответа порядок старой
   страницы неизменен.
4. **Структурно**: `tableDataSource.sort === null` в серверном режиме,
   привязан к `MatSort` в клиентском.
5. **Регрессия клиентского режима**: клик сортирует локально.
6. **Регрессия legacy manual**: `paginatorState` без `mode`/`pageSource` → sort привязан
   (локальная сортировка сохранена).
