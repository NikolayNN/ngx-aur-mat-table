# Начальное состояние сортировки: табличный `sortCfg` (active + direction)

**Дата:** 2026-06-12
**Статус:** Design (одобрен к реализации)
**Тип изменения:** аддитивная фича, новый опциональный конфиг `TableConfig.sortCfg`.
Без `sortCfg` поведение не меняется (закреплено регрессионным тестом).
**Контекст батча:** пункт 2 фидбека от 2026-06-11 (Andrey Patsko); ветка `feat/19.7.0-feedback`,
коммит на пункт. Опирается на пункт 1 (server sort mode): в серверной обвязке `MatSort`
не привязан к dataSource.

## Проблема

Начальную сортировку задать нельзя. `startServerController()` читает `matSort?.active`
(`ngx-aur-mat-table.component.ts:798`), а он пуст до первого клика — первый серверный
запрос уходит без `sort`, стрелка в заголовке не горит. Потребителю нужно: первый запрос
сразу с `{active: 'id', direction: 'desc'}` и согласованная стрелка. В клиентском режиме
начальный порядок тоже не задаётся декларативно.

## Решение (зафиксировано с пользователем)

| Решение | Выбор |
|---|---|
| Механизм | биндинги `[matSortActive]`/`[matSortDirection]` на `<table>`; НЕ императивный `matSort.sort()` (эмитит `sortChange` → дубль-запрос) и НЕ пер-колоночный initial (конфликт нескольких колонок) |
| Где живёт конфиг | табличный `TableConfig.sortCfg?: TableSortConfig` (имя `SortConfig` занято колоночным); сюда же при необходимости позже ляжет `mode` (решение пункта 1) |
| Обязательность | `sortCfg` опционален — без него поведение ровно текущее; внутри конфига оба поля обязательны (`active` + `direction`), половинчатого состояния нет |
| Семантика | начальное состояние, НЕ реактивный контрол: смена в рантайме подвинет стрелку, но не пересортирует данные и не вызовет запрос |
| Клиентский режим | поддерживается (бесплатно через `_sort.initialized` в `MatTableDataSource`), хотя фидбек просил только серверный кейс — единообразие |

## Контракт — `model/ColumnConfig.ts`

```ts
/**
 * Начальная сортировка таблицы: стрелка в заголовке + начальный порядок.
 * Это начальное состояние, а не реактивный контрол: смена значения в рантайме
 * передвинет стрелку, но не пересортирует данные и не вызовет серверный запрос.
 */
export interface TableSortConfig {
  /**
   * Ключ колонки из `columnsCfg`, у которой включён `sort`.
   * Ключ колонки без `sort` (или несуществующий) стрелку не покажет,
   * но в клиентском режиме данные по нему всё равно отсортируются.
   */
  active: string;
  /** Направление начальной сортировки. */
  direction: 'asc' | 'desc';
}

export interface TableConfig<T> {
  // ...
  /** Начальная сортировка. Не задан — без начальной сортировки (текущее поведение). */
  sortCfg?: TableSortConfig;
}
```

`TableSortConfig` экспортируется из `public-api.ts` (как остальные конфиг-интерфейсы).

## Реализация — `ngx-aur-mat-table.component.html`

Два биндинга на `<table mat-table matSort ...>`:

```html
[matSortActive]="tableConfig.sortCfg?.active ?? ''"
[matSortDirection]="tableConfig.sortCfg?.direction ?? ''"
```

Код компонента не меняется:

- серверный режим: `startServerController()` уже читает `matSort.active` —
  директива инициализирована до `ngAfterViewInit` (`static: true`), первый запрос
  уходит с сортировкой автоматически;
- клиентский режим: `MatTableDataSource` применяет начальное состояние при привязке
  (merge `sortChange` + `_sort.initialized`);
- инициализация `MatSort` входами НЕ эмитит `matSortChange` → нет дубль-запроса
  и шума в `(sort)`.

Нюанс корректности: Angular переприсваивает input только при смене значения биндинга,
поэтому после клика пользователя константный `sortCfg` не перезатирает состояние директивы.

## Поведение по режимам

| Режим | Стрелка | Данные / первый запрос |
|---|---|---|
| `pageSource` | горит сразу | первый запрос с `sort` из конфига; запрос ровно один |
| `mode: 'server'` ручной (пункт 5) | горит | хост сам строит первый запрос из своего же конфига; событие не нужно |
| Клиентский | горит | данные отсортированы при инициализации |
| Без `sortCfg` | нет (как сейчас) | как сейчас (регрессионный пин) |

## Edge cases

- **`active` на колонку без `sort`/с опечаткой**: заголовок не `MatSortable` → стрелки нет;
  в клиентском режиме dataSource всё равно отсортирует по этому ключу (Material не сверяется
  с заголовками). Кодом не валидируем (YAGNI), предупреждение в JSDoc.
- **Взаимодействие с пунктом 1**: в серверной обвязке sort отвязан от dataSource →
  начальная сортировка не пересортирует серверную страницу локально.
- **Toggle после initial**: `MatSort` продолжает свой стандартный цикл от начального
  состояния (дефолтный цикл `asc` → `desc` → сброс `''`). Значит, после initial `desc`
  следующий клик даёт **сброс** (`direction: ''`) и запрос уйдёт с ним — это штатное
  поведение Material, не баг. Тест закрепляет продолжение цикла на варианте
  initial `asc` → клик → `desc`.
- **Runtime-смена `sortCfg`**: стрелка подвинется (смена значения биндинга), данные/запросы
  не тронутся — документированная семантика; то же ограничение config-only-change, что и у
  остальных конфигов.
- **`customSort`** в клиентском режиме применяется к начальной сортировке как обычно
  (через `sortingDataAccessor`).

## Тесты — новый `ngx-aur-mat-table-initial-sort.spec.ts`

1. **Server, первый запрос**: `sortCfg: {active: 'name', direction: 'desc'}` → первый вызов
   `pageSource` содержит этот `sort`, и вызов ровно один (без дублей от инициализации).
2. **Server, стрелка**: сразу после инициализации `th` сортируемой колонки имеет
   `aria-sort="descending"`.
3. **Server, порядок**: строки отображаются ровно в порядке серверного ответа
   (initial sort не включает локальную пересортировку — взаимодействие с пунктом 1).
4. **Client**: данные начально отсортированы по `sortCfg` (DOM-порядок) + стрелка горит.
5. **Регрессия без `sortCfg`**: первый запрос `pageSource` уходит с `sort: undefined`;
   стрелки нет.
6. **Toggle после initial**: `sortCfg: {active: 'name', direction: 'asc'}`, клик по этой
   колонке → направление `desc` (цикл продолжается от начального состояния, а не с нуля),
   уходит новый запрос с `desc` и `pageIndex: 0`.
