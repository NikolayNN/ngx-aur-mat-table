# Стабилизация action-колонок через `trackBy` (фикс устаревших/пустых ячеек действий)

**Дата:** 2026-06-29
**Статус:** Design (одобрен к реализации)
**Тип изменения:** Fix (исправление двух багов рендера). Публичный API не меняется,
существующее поведение не ломается.
**Контекст батча:** Spec A из батча отзывов по `actionCfg` (сбор от 2026-06-29).
Закрывает отзывы **#1** (динамические иконки не обновляются после смены состояния
строки) и **#3** (ячейки action-колонки пусты при async-первой загрузке) — у них
**один корень**. Отзыв **#2** (визуальный disabled-стиль иконки) вынесен в Spec B
(следующая работа того же батча). Оба спека целятся в один minor-бамп.

---

## Проблема

В таблицах action-кнопки задаются резолверами от строки (`action`, `icon.name`,
`icon.color`, `icon.tooltip`, `visible`, `disabled`, пункты меню). После успешного REST
хост подменяет `[tableData]` **новым массивом с новыми объектами строк** (например, у
строки `enabled: false → true` или `isDefault: false → true`).

**Ожидание:** action-колонка пересчитывает все row-зависимые свойства по новым данным —
`toggle_off → toggle_on`, `star_border → star`, новый цвет/tooltip/код действия.

**Факт:**
- **#1** — REST проходит, данные строки обновлены, но иконка визуально остаётся
  в старом состоянии.
- **#3** — таблица впервые рендерится с пустым `tableData`; затем хост асинхронно
  передаёт непустой массив; action-колонка появляется, но её ячейки остаются пустыми
  (в DOM `<td>` есть, кнопок нет).

### Корень (проверено по коду)

Перерасчёт **не виноват**: `RowActionProvider.setView()` →
`ActionViewFactory.create()` корректно строит `actionView` из новых строк
(`factories/ActionViewFactory.ts:11-33` — резолвит `action`, `icon.*`, `visible`,
`disabled`, `menu`). Виноват **рендер**.

Action-колонки рисуются так (`ngx-aur-mat-table.component.html:242`):

```html
<ng-container *ngFor="let col of rowActionsProvider.columns" [matColumnDef]="col.columnName">
```

а `rowActionsProvider` **пересоздаётся новым экземпляром на каждую смену данных**
(`ngx-aur-mat-table.component.ts:594`):

```ts
this.rowActionsProvider = RowActionProvider.create(this.tableConfig)
  .addActionColumns(this._displayColumns)
  .setView(this.tableDataSource.data);
```

У этого `*ngFor` **нет `trackBy`**. Когда массив `columns` получает новые идентичности
объектов, Angular уничтожает старый `MatColumnDef` action-колонки и создаёт новый — но
`MatTable` **не перерисовывает уже материализованные ячейки** под новый def. Отрисованные
action-ячейки продолжают держать ссылку на **старый `col`** (его контекст embedded-view),
чей `actionView` посчитан по **старому/пустому** состоянию строк.

- **#1:** новые строки (`rowId` позиционный, 0…N−1) ищут `staleCol.actionView.get(rowId)`
  и получают действия, посчитанные по прежнему состоянию → старая иконка/цвет/действие.
- **#3:** первый рендер с пустыми данными регистрирует def с пустым `actionView`. Когда
  данные приходят, `initTable()` присваивает `tableDataSource.data`
  (`ngx-aur-mat-table.component.ts:655`) и `MatTable` материализует строки
  **синхронно**, но новый провайдер (с заполненным `actionView`) присваивается позже
  (`:594`), а замена def в `*ngFor` — ещё позже, в CD. Строки материализуются против
  старого (пустого) `col`, и без `trackBy` поздняя замена их не перерисовывает.

Зафиксированное в проде состояние (у провайдера `columns[0].actionView` содержит ключи
0…11, а DOM-ячейки пусты) — точный отпечаток «ячейки привязаны к предыдущему экземпляру
`col`».

### Почему `cdr.markForCheck()` не помогает

Проблема не в «непомеченной» вьюхе, а в **устаревшей ссылке** `col` в замыкании
отрисованной ячейки. CD честно перевычисляет `col.actionView.get(element.rowId)`, но `col`
— старый объект с пустым/старым `actionView`. Пока def пересоздаётся (а ячейки — нет),
никакой `markForCheck` это не лечит. Гипотеза автора отзыва верна.

Замечание: только action-колонки уязвимы — они рисуются через `*ngFor` по массиву
(поддержка нескольких action-колонок). Колонки индекса и выделения используют `*ngIf` +
статическое `COLUMN_NAME`, поэтому их def стабильны и багу не подвержены.

## Решение (F1 — `trackBy`, выбрано с пользователем)

Стабилизировать def через `trackBy` по `col.columnName`. Имя колонки **стабильно и
уникально**: `RowActionProvider.resolveConfigs` (`providers/RowActionProvider.ts:46-65`)
дедупит/отбрасывает коллизии ключей (dev-warn), поэтому ключ — корректный идентификатор
для трекинга.

Со стабильным ключом `NgForOf` переиспользует embedded-view и тот же `MatColumnDef`,
обновляя контекст `col` **на месте** на новый объект. Уже отрисованные ячейки `MatTable`
резолвят `col` через эту переиспользуемую declaration-view, поэтому на **том же CD-проходе**
читают свежий `actionView` — иконки обновляются (#1), а ячейки первой async-загрузки
наполняются (#3).

### Отвергнутая альтернатива (F2)

Держать экземпляр `RowActionProvider` стабильным (пересоздавать только при смене ссылки
`tableConfig`), обновляя `actionView` на месте. Устраняет teardown в корне, но добавляет
спец-случай в жизненный цикл провайдера (action-провайдер вёл бы себя иначе, чем все
остальные, которые пересобираются на каждую загрузку) при пренебрежимом выигрыше на
таблицах ≤1000 строк. Отклонено в пользу хирургического F1.

## Изменения (2 файла)

### 1. `projects/ngx-aur-mat-table/src/lib/ngx-aur-mat-table.component.ts`

Расширить импорт `RowActionProvider`, добавив `ActionColumnView`:

```ts
import {ActionEvent, ActionColumnView, RowActionProvider, RowActionProviderDummy} from './providers/RowActionProvider';
```

Добавить метод рядом с `trackByRow` (≈ строка 835):

```ts
/** trackBy для *ngFor по action-колонкам: columnName стабилен/уникален
 *  (RowActionProvider дедупит ключи), поэтому MatColumnDef переиспользуется между
 *  сменами [tableData] — пересозданный провайдер обновляет col на месте, а не
 *  пересоздаёт def (иначе уже отрисованные ячейки держат старый col → устаревший
 *  actionView). Фикс #1 (иконки не обновляются) и #3 (пустые ячейки при async-загрузке). */
trackByActionColumn = (_: number, col: ActionColumnView<T>): string => col.columnName;
```

### 2. `projects/ngx-aur-mat-table/src/lib/ngx-aur-mat-table.component.html` (строка 242)

```html
<!-- action columns (одна или несколько) -->
<ng-container *ngFor="let col of rowActionsProvider.columns; trackBy: trackByActionColumn"
              [matColumnDef]="col.columnName">
```

Остальная разметка action-колонки не меняется.

## Тесты

Новый `projects/ngx-aur-mat-table/src/lib/ngx-aur-mat-table-action-rerender.spec.ts`,
по образцу `ngx-aur-mat-table-action-disabled.spec.ts` (host-компонент с
`[tableConfig]`/`[tableData]`, `NgxAurMatTableModule` + `NoopAnimationsModule`, селекторы
`tr.mat-mdc-row` / `button` / `mat-icon`, описания на русском).

1. **#1 — динамическое обновление по смене данных.** Action с зависимыми от строки
   `icon.name` (`enabled ? 'toggle_on' : 'toggle_off'`), `icon.color` (GREEN/GRAY) и
   `action` (`enabled ? 'disable' : 'enable'`). После `detectChanges()` проверить
   стартовый текст `<mat-icon>` и `style.color`. Затем присвоить `host.data` **новому
   массиву новых объектов** с перевёрнутым `enabled`; `detectChanges()`. Проверить, что
   текст иконки (`toggle_off → toggle_on`), цвет и код действия (по клику) отражают новое
   состояние.

2. **#3 — async-первая загрузка.** Старт с `tableData = []` → в action-ячейках нет
   `button`. Присвоить непустой `tableData`; `detectChanges()`. Проверить, что кнопки
   появились для каждой строки (ожидаемое число `button[mat-icon-button]` в action-ячейках).

3. **Несколько action-колонок (страж уникальности).** Две action-колонки с разными `key`.
   После смены данных обновляются обе колонки — фиксирует корректность ключа `trackBy`.

Сами тесты — красно-зелёным в фазе реализации (TDD): сначала падают на текущем коде
(особенно #1/#3), затем зеленеют после правок. Так субтильная зависимость от семантики
`NgForOf` (обновление контекста на месте) остаётся зафиксированной регрессионным тестом.

## Edge cases

- **Смена структуры колонок** (число/имена/позиции) происходит только при смене
  `tableConfig`, не данных. `trackBy` по `columnName` корректно обрабатывает
  добавление/удаление/перестановку (создаёт/уничтожает/двигает def по ключу).
- **`RowActionProviderDummy`** имеет пустой `columns` → `*ngFor` ничего не рисует,
  `trackByActionColumn` не вызывается. Безопасно.
- **Ортогональность trackBy.** `trackByActionColumn` (переиспользование def колонки) и
  табличный `trackByRow` (переиспользование строк) независимы и не взаимодействуют.
- **Транзиент первой загрузки.** Строки кратко материализуются против старого пустого
  `col` во время синхронного `.data =`, затем восстанавливаются в **том же** CD-проходе
  (после переприсвоения провайдера и обновления `col` на месте в `*ngFor`), до отрисовки
  кадра. Видимого мерцания нет.
