# 19.16.0

Breaking refactor: `paginationCfg.mode` is now the single mode switch. Manual server
pagination (`mode:'server'` + `[paginatorState]` + `(pageChange)`) is fixed — in 19.15.0
the paginator was still bound to `MatTableDataSource`, causing client-side re-slicing of
the already-paged server page (empty pages, spurious `pageChange`, pageIndex reset to 0).
Suite green, library and demo build clean.

## Fixed: manual server pagination — `mode:'server'` + `[paginatorState]` + `(pageChange)` no longer re-slices the server page

In 19.15.0, `mode:'server'` without a `pageSource` left the paginator bound to
`MatTableDataSource`. The data source's internal `_updatePaginator()` then re-sliced the
already-paged server slice: it overwrote `paginator.length` with the array length, clamped
`pageIndex` back to 0 on pages beyond the first, and emitted a spurious `(pageChange)` on
every load. In effect the host's 20-row server page was truncated to zero visible rows on
any page past the first.

The fix is that all wiring (paginator binding, sort binding, index offset, timeline
slicing) is now controlled by a single predicate: `mode === 'server'` or `!!pageSource`.
With `mode:'server'`, the paginator is set to `null` so `MatTableDataSource` never
re-slices; the host's data is rendered verbatim, `(pageChange)` emits correctly, and
server-side sort is preserved.

## Changed (breaking): `paginationCfg.mode` is the single mode switch

`paginatorState` is now **state only** — it carries `total` and `pageIndex` for the
paginator UI, but no longer activates server mode. Previously, mere presence of
`[paginatorState]` without `mode` triggered a contradictory half-server state (sort stayed
client; index offset and timeline were server; pagination was broken). That combination is
now fully **client**.

| Configuration | Behavior |
|---|---|
| `mode: 'client'` or unset | Client pagination + sort in memory |
| `mode: 'server'` + `[paginatorState]` + `(pageChange)` | Manual server — host loads pages; table never re-slices |
| `mode: 'server'` + `[pageSource]` | Auto server — table owns the load loop (unchanged) |
| `[pageSource]` without `mode` | Implies server (unchanged) |
| bare `[paginatorState]` without `mode` | **Client** (breaking — was contradictory half-server) |

Migration: if you rely on server behavior driven only by `[paginatorState]` (no `mode`),
add `mode: 'server'` to `paginationCfg`. See `docs/MIGRATION-19.16.0.md`.

## Removed (breaking): `new PaginatorState(total, pageIndex)` positional constructor

The public positional constructor is now `private`. Use the named factories:

```ts
// before
new PaginatorState(total, pageIndex)
// after
PaginatorState.of({ total, pageIndex })
// or for an empty/default state
PaginatorState.empty()
```

## Removed (breaking): `NgxAurTablePageEventUtils.createEmpty(cfg)`

The deprecated utility is removed. In server mode the table auto-seeds an empty
`PaginatorState` on `ngOnInit`, so no explicit call is needed. For cases where an explicit
empty state is required, use `PaginatorState.empty()` directly.
