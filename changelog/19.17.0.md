# 19.17.0

Two `actionCfg` rendering fixes, no public API change. Action columns now recompute
correctly when `[tableData]` changes, and disabled action/menu icons render with a neutral
disabled color instead of their configured `icon.color`. Suite green.

## Fixed: action columns no longer show stale or empty cells after `[tableData]` changes

Row actions are computed per row (`action`, `icon.name`/`color`/`tooltip`, `visible`,
`disabled`, menu items). When the host replaced `[tableData]` with a new array of new row
objects, the action cells kept rendering the previous state — and on an async first load
(empty array, then a populated one) the action cells stayed empty. The icon, color,
tooltip and emitted action code did not update.

Action columns are rendered with `*ngFor` over the row-action provider's columns, and the
provider is rebuilt on every data change. Without a `trackBy`, Angular tore down and
recreated each action `MatColumnDef`, but `MatTable` does not re-render already-materialized
cells against the new definition, so the cells stayed bound to the previous column object
(with stale or empty per-row data). Adding `trackBy` keyed by the column name lets
`NgForOf` reuse the column definition and update its context in place, so the cells pick up
the fresh per-row action view on the same change-detection pass.

## Fixed: disabled action and menu icons render with a neutral disabled color

A disabled action kept its configured `icon.color` (e.g. a blue copy or red delete), so the
inline color overrode Material's disabled styling and the button looked active even though
it was not clickable. Disabled direct-action buttons, menu-trigger buttons and menu items
now drop the inline color while disabled, so the icon inherits Material's neutral disabled
color. No `icon.color` change or `::ng-deep` is needed on the consumer side; enabled
actions are unaffected.
