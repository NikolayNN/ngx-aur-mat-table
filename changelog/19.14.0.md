# 19.14.0

Bug fix with a small API migration: a column whose `key` is `'id'` used to silently
overwrite the table's internal row index, so `valueView` icons, row styles, the index
column and the timeline misbehaved on any table that also had an `id` column. The
internal index moved off the public `id` field to **`rowId`**, so `key: 'id'` is now an
ordinary business column. Suite green (272 tests), library and demo build clean.

**Migration:** if you read the row *index* from a row object — `row.id` in a template
context, or via the exported `TableRow` type — use the directive context's **`index`**
instead. It is unchanged and still gives the row index. Reading `row.id` now returns
`undefined` (or your `id` column's business value, if you configure one). Reads of your
own business `id` field via `valueConverter`, `trackBy`, etc. are unaffected — only the
library's internal `TableRow.id` was renamed.

## Fixed: `key: 'id'` column no longer breaks valueView / row styles / index / timeline

`TableRow` stored its positional row index in a public field named `id`. The library
overlays each column's value onto the row by key (`row[key] = valueConverter(obj)`), so
a column configured with `key: 'id'` overwrote that index with the business value. Every
positional lookup then missed: `valueView` cells fell back to plain text (or showed
another row's view), the index column printed the business id, and row styles and the
timeline gap detection broke. Business ids that happened to fall inside the `0..N-1`
index range rendered "by luck"; larger ids did not — the classic "some rows show icons,
others show text" symptom.

The internal index is now stored on **`rowId`**, so `key: 'id'` is a normal column
again. The public `index` exposed to `ngxAur*Def` template contexts is preserved (now
sourced from `rowId`), so `let-i="index"` bindings are unchanged.

## Added: guard against column keys colliding with reserved row fields

To prevent the same class of silent corruption, configuring a column whose `key` is a
reserved `TableRow` field name (`rowId` or `rowSrc`) now throws at init with a clear
message, the same way a missing `[tableConfig]` does. Rename the column's `key` and read
the value through its `valueConverter`.
